# 현실에 가까운 수송 시뮬레이션을 위한 요구사항 정리

## 목적

이 문서는 현재 전시 예비군 수송 시뮬레이션을 더 현실적인 연구·논문 수준의 실험으로 발전시키기 위해 필요한 자료, 모형 개선, 검증 절차를 정리한 것이다.

현재 모형은 버스 단독 수송과 철도-버스 복합 수송의 구조적 차이를 비교할 수 있는 baseline으로는 의미가 있다. 그러나 실제 작전계획이나 논문 최종 제출 수준의 현실성을 확보하려면, 추상 네트워크와 가정값을 실제 자료 또는 명시적 근거가 있는 시나리오로 보정해야 한다.

핵심 목표는 다음과 같다.

- 실제 수송 환경과 더 가까운 네트워크를 사용한다.
- 송파구뿐 아니라 다른 지역에도 재사용 가능한 입력·네트워크 구축 방식을 만든다.
- 차량, 철도, 환승, 도로 혼잡, 장애 확률을 근거 있는 값으로 보정한다.
- 결과가 특정 가정 하나에만 의존하지 않는지 확인한다.
- 결과를 재현 가능하고 검증 가능한 형태로 정리한다.
- 보고서 결론이 “모형 안에서의 조건부 비교”인지, “현실적 운용 판단에 가까운 결과”인지 명확히 구분한다.

## 현재 모형의 위치

현재 모형은 다음 성격을 가진다.

- 현실적 수송 요소를 포함한 추상 네트워크 기반 실험
- 약 1,000명 규모의 수송 비교
- 버스 단독 수송과 철도-버스 복합 수송 비교
- 차량 대수, 배차, 회차, 철도 운행 간격, 환승 시간, 지각, 도로 혼잡, 도로 장애 반영
- 도착하지 못한 인원과 자원 효율을 함께 평가

따라서 현재 결과는 단순 장난감 모델은 아니다. 다만 실제 지도, 실제 도로 용량, 실제 철도 운행자료, 실제 군 수송 절차로 보정된 결과는 아니다.

현재 결과의 적절한 해석은 다음과 같다.

> 현재 가정한 대표 네트워크와 운용 조건에서는 버스 단독 수송이 시간상 유리하고, 복합 수송은 자원 효율에서 장점이 있으나 철도 전후 도로 구간의 취약성에 민감하다.

현재 결과를 다음처럼 말하면 과장이다.

> 실제 송파구에서 전방 목적지까지 이동할 때 버스 단독 수송이 항상 더 빠르다.

## 현실성 확보를 위한 핵심 요구사항

현실에 가까운 시뮬레이션을 위해서는 크게 여덟 가지 축이 필요하다.

1. 실제 또는 준실제 교통 네트워크
2. 현실적인 수송 수요와 집결 행태
3. 차량과 운전 인력 가용성
4. 실제적인 철도 운행 조건
5. 도로 혼잡과 통행시간 보정
6. 장애·위협·차단 시나리오 보정
7. 환승·대기·지휘통제 시간 보정
8. 검증, 민감도 분석, 재현성 문서화
9. 지역 재사용성과 공개 지도·오픈소스 활용
10. 자료·보안·해상도 제약 안에서의 현실성 확보

아래 항목들이 충족될수록 결과는 단순 비교 실험에서 현실적인 정책·운용 판단에 가까워진다.

## 1. 실제 또는 준실제 네트워크

가장 중요한 보강은 네트워크다. 현재처럼 몇 개의 대표 노드와 링크로 구성된 네트워크는 구조 비교에는 유용하지만, 실제 운용 판단에는 한계가 있다.

필요한 네트워크 요소는 다음과 같다.

| 필요 요소 | 설명 | 중요한 이유 |
|---|---|---|
| 실제 출발지 후보 | 송파구 내 집결지 또는 동원 장소 후보 | 출발 위치가 달라지면 접근 도로와 초기 혼잡이 달라진다. |
| 실제 목적지 또는 목적지 권역 | 전방 목적지 또는 목적지 권역 | 마지막 도로 구간의 취약성이 결과를 크게 바꾼다. |
| 실제 도로망 | 주요 간선도로, 고속도로, 우회도로, 병목 구간 | 버스 단독 수송의 성과는 도로망 구조에 직접 의존한다. |
| 실제 철도 접근 거점 | 사용할 수 있는 역, 화물·군 수송 가능 거점, 환승 위치 | 복합 수송은 철도 자체보다 철도 접근성이 중요하다. |
| 철도 이후 도로망 | 도착역에서 전방 목적지까지의 마지막 도로망 | 현재 결과에서 복합 수송 취약성이 크게 나타난 구간이다. |
| 우회 가능성 | 주요 도로 장애 시 대체 경로 | 장애 상황에서 수송 방식의 안정성을 결정한다. |

현실적인 네트워크에는 최소한 다음 정보가 포함되어야 한다.

- 링크별 길이
- 평시 또는 기준 통행시간
- 차로 수 또는 용량 추정값
- 도로 종류
- 주요 병목 지점
- 우회 가능 경로
- 철도역과 도로 연결부
- 목적지 직전 마지막 접근로

현실성을 높이는 가장 좋은 방식은 실제 지도 기반 네트워크를 사용하고, 민감한 군사 목적지는 권역 단위로 익명화하는 것이다.

## 1-A. 지역 확장성과 공개 지도·오픈소스 활용

추후 송파구뿐 아니라 다른 지역에 대해서도 같은 방식으로 시뮬레이션을 돌리려면, 네트워크를 손으로 매번 새로 만드는 방식은 적합하지 않다. 지역이 바뀌어도 재사용할 수 있는 네트워크 생성 절차가 필요하다.

핵심 방향은 다음과 같다.

- 지역 이름, 행정구역, 좌표, 또는 경계 파일을 입력하면 도로망을 자동으로 가져온다.
- 도로망에서 출발지, 철도 접근 거점, 목적지 후보를 연결한다.
- 도로 속성, 길이, 도로 등급, 교차 구조를 최대한 자동으로 추출한다.
- 지역별 수송 시나리오는 별도 설정 파일로 분리한다.
- 민감한 목적지는 실제 좌표 대신 권역 또는 가상 노드로 대체할 수 있게 한다.

활용 가능성이 높은 공개 자료와 도구는 다음과 같다.

| 범주 | 활용 대상 | 역할 |
|---|---|---|
| 공개 지도 | OpenStreetMap | 도로망, 도로 종류, 교차 구조, 대략적 경로 추출 |
| 도로 네트워크 도구 | OSMnx, NetworkX | 지역별 도로망 다운로드, 그래프 변환, 최단경로 계산 |
| 공간 자료 처리 | GeoPandas, Shapely | 행정구역, 좌표, 권역, 노드·링크 공간 처리 |
| 경로·시간 보정 | 공개 교통 API 또는 지도 API | 기준 통행시간, 거리, 경로 검증 |
| 철도 자료 | 공개 역 위치, 노선 자료, 시간표 자료 | 철도 접근성, 운행 간격, 환승 후보 구성 |
| 공개 GitHub repo | OSMnx 예제, traffic assignment 예제, SimPy transportation examples | 네트워크 구축, 교통배정, 시뮬레이션 구조 재사용 |

공개 GitHub repo를 활용할 때의 목적은 “남의 코드를 그대로 붙이는 것”이 아니라, 다음 기능을 검증된 방식으로 재사용하는 것이다.

- 지도 데이터 불러오기
- 도로망을 그래프로 변환하기
- 도로 등급별 속도와 용량 추정하기
- 최단경로 또는 시간기반 경로 찾기
- 여러 출발지·도착지 조합을 자동 생성하기
- 반복 실험을 재현 가능하게 관리하기

지역 재사용성을 높이려면 프로젝트 구조도 다음처럼 분리되는 것이 좋다.

| 구성 요소 | 역할 |
|---|---|
| 지역 입력 파일 | 분석할 지역, 출발지 후보, 철도 거점 후보, 목적지 권역 정의 |
| 네트워크 생성기 | 지도 자료에서 도로·철도 네트워크 생성 |
| 모수 보정 파일 | 지역별 도로 속도, 용량, 혼잡, 차량 가용성 설정 |
| 시나리오 파일 | 인원 규모, 장애 조건, 수송 정책 설정 |
| 결과 비교 모듈 | 지역이 달라도 같은 지표로 결과 비교 |

이렇게 만들면 송파구 사례는 하나의 case study가 되고, 다른 지역은 같은 파이프라인에 입력값만 바꿔 넣는 방식으로 확장할 수 있다.

## 1-B. 공개 지도 기반 접근의 한계

공개 지도와 오픈소스 도구를 사용하면 재사용성은 커지지만, 그 자체로 현실성이 완성되는 것은 아니다.

주의해야 할 한계는 다음과 같다.

- 공개 지도에는 실제 차로 수, 군 통제 가능성, 검문 지연, 전시 교통통제 정보가 부족할 수 있다.
- 도로 길이는 잘 맞아도 실제 통행시간은 시간대와 혼잡에 따라 크게 달라진다.
- 철도역 위치는 알 수 있어도 군 수송 가능 여부나 열차 편성 가능성은 별도 자료가 필요하다.
- 민감한 목적지를 실제 좌표로 쓰면 보안 문제가 생길 수 있다.
- 공개 지도 품질은 지역마다 다르다.
- 실제 작전 중 도로 우선권, 통제, 대피 교통은 공개 자료만으로 추정하기 어렵다.

따라서 공개 지도 기반 네트워크는 “현실에 가까워지는 출발점”이지 “현실 그 자체”가 아니다. 현실성을 높이려면 공개 지도 위에 보정값과 제약 조건을 추가해야 한다.

## 2. 수송 수요와 집결 행태

현재는 약 1,000명이 하나의 집결지에서 출발하는 상황을 가정한다. 현실적으로는 인원이 여러 지점에서 모이거나, 도착 시간이 더 넓게 퍼지거나, 일부 인원이 지연될 수 있다.

필요한 보정 요소는 다음과 같다.

| 필요 요소 | 설명 | 중요한 이유 |
|---|---|---|
| 인원 규모의 범위 | 500명, 1,000명, 2,000명 등 여러 규모 | 결론이 특정 인원 규모에만 맞는지 확인할 수 있다. |
| 집결지 수 | 단일 집결지 또는 다중 집결지 | 다중 집결지는 초기 교통량과 차량 회차를 바꾼다. |
| 도착 시간 분포 | 정시 도착, 완만한 지각, 장기 지각 꼬리 | 출발 정책의 효과를 평가하는 핵심 요소다. |
| 탑승 절차 시간 | 명단 확인, 장비 적재, 탑승 통제 | 실제 출발 지연을 만든다. |
| 결원 또는 미집결률 | 일정 비율의 미집결 가능성 | 완료율과 수송 목표 해석에 영향을 준다. |

논문 수준에서는 “왜 이 도착 분포를 사용했는지”가 설명되어야 한다. 실제 자료가 없으면 전문가 판단, 유사 동원훈련 경험, 보수적 시나리오 등으로 근거를 제시해야 한다.

## 3. 차량과 운전 인력 가용성

차량 대수와 회차 시간은 수송 성과를 크게 좌우한다. 현재 모형에는 차량 수, 배차 간격, 회차 시간이 포함되어 있지만, 이 값이 실제 가용 자원과 맞는지는 별도 검증이 필요하다.

필요한 정보는 다음과 같다.

- 실제 투입 가능한 버스 대수
- 차량당 탑승 가능 인원
- 운전 인력 가용성
- 차량 준비 시간
- 집결지 진입·대기·출차 시간
- 목적지 하차 시간
- 회차 가능 여부
- 회차 시 필요한 연료, 정비, 휴식 조건
- 전시 상황에서 민간 버스 동원 가능성

특히 “버스 23대” 같은 값은 현실적인 설명이 필요하다. 예를 들어 1,000명을 45명씩 나누면 약 23대가 필요하다는 계산은 자연스럽지만, 실제로 23대가 동시에 확보 가능한지는 별개의 문제다.

현실화된 실험에서는 차량 조건을 단일값으로 고정하기보다 다음처럼 여러 수준으로 나누는 것이 좋다.

- 낙관 조건: 충분한 차량과 운전 인력 확보
- 기준 조건: 계획상 확보 가능한 차량
- 제약 조건: 일부 차량 지연 또는 부족
- 악화 조건: 회차 지연, 연료·정비·운전 인력 제약

## 4. 철도 운행 조건

복합 수송을 현실적으로 평가하려면 철도 구간을 단순히 “빠르고 대량 수송 가능한 링크”로만 보면 안 된다. 실제 철도 운행 가능성, 편성, 승하차, 선로 우선권, 역 처리 능력이 중요하다.

필요한 정보는 다음과 같다.

| 필요 요소 | 설명 |
|---|---|
| 사용 가능한 철도역 | 출발 측 접근역과 도착 측 하차역 |
| 열차 편성 가능성 | 객차 수, 수송 가능 인원, 군 수송 가능 여부 |
| 운행 간격 | 평시 간격이 아니라 위기 상황에서 확보 가능한 간격 |
| 선로 용량 | 일반 열차와 군 수송 열차가 같은 선로를 쓰는지 여부 |
| 승하차 처리 능력 | 플랫폼 동선, 대기 공간, 질서 유지 가능성 |
| 철도 장애 가능성 | 선로, 전력, 신호, 역 시설 장애 |
| 철도 우선권 | 군 수송이 기존 열차보다 우선될 수 있는지 여부 |

현재 모형은 철도가 정상 운행된다는 가정에 가깝다. 현실성을 높이려면 철도에도 장애 또는 지연 시나리오를 넣어야 한다.

예를 들어 다음 시나리오가 필요하다.

- 철도 정상 운행
- 열차 간격 증가
- 일부 편성 축소
- 도착역 처리 지연
- 철도 접근 도로 장애
- 철도 하차 후 마지막 도로 구간 장애

복합 수송의 핵심은 철도 자체가 아니라 “철도 전후 연결부”다. 따라서 철도 구간만 정교하게 만들고 접근·환승·마지막 구간을 단순화하면 현실성이 부족하다.

## 5. 도로 혼잡과 통행시간 보정

현재 모형은 교통량이 늘어나면 통행시간이 증가하는 구조를 반영한다. 하지만 실제 도로 혼잡을 더 정확하게 반영하려면 도로별 기준 통행시간, 용량, 배경 교통량이 필요하다.

필요한 자료는 다음과 같다.

- 시간대별 평시 통행시간
- 도로별 용량 또는 차로 수
- 배경 교통량
- 주요 병목 구간
- 신호 교차로 또는 램프 영향
- 군 수송 차량이 일반 교통과 섞이는 정도
- 버스 대열 운행 시 실제 속도 저하
- 경찰 또는 군 통제에 따른 통행 우선권 가능성

특히 전시 상황에서는 평시 통행시간이 그대로 유지되지 않는다. 주민 이동, 대피, 통제, 사고, 도로 손상, 검문, 교통 통제 등이 함께 발생할 수 있다.

현실화된 실험은 다음 세 종류의 교통 조건을 나누어야 한다.

- 평시 기준 조건
- 위기 상황 혼잡 조건
- 장애·통제 포함 악화 조건

논문 수준에서는 “도로 혼잡 배율”을 단순히 임의로 두기보다, 왜 그 배율이 합리적인지 설명해야 한다.

## 6. 장애와 차단 시나리오

현재 모형은 도로의 완전 차단과 용량 저하를 반영할 수 있다. 이는 중요한 개선점이다. 다만 현실성을 높이려면 장애 확률과 장애 위치가 더 근거 있게 설정되어야 한다.

필요한 보정 요소는 다음과 같다.

| 필요 요소 | 설명 |
|---|---|
| 장애 유형 | 완전 차단, 차로 일부 폐쇄, 속도 저하, 검문 지연, 사고 |
| 장애 위치 | 교량, 터널, 램프, 역 접근로, 마지막 접근로 |
| 장애 확률 | 도로 종류나 중요도에 따른 차등 확률 |
| 장애 지속 시간 | 일시 장애인지 장시간 장애인지 |
| 복구 가능성 | 우회, 복구, 통제 전환 가능 여부 |
| 상관관계 | 같은 지역의 여러 도로가 동시에 영향을 받을 가능성 |

현실적으로는 도로 장애가 완전히 독립적으로 발생한다고 보기 어렵다. 특정 권역이 영향을 받으면 인접 도로도 함께 혼잡해지거나 차단될 수 있다. 따라서 고도화된 실험에서는 장애의 공간적 상관관계를 고려하는 것이 좋다.

중요한 시나리오는 다음과 같다.

- 주요 간선도로 1개 차단
- 철도 접근 도로 차단
- 마지막 도로 구간 차단
- 버스 우회로 일부 차단
- 여러 도로의 동시 용량 저하
- 철도는 정상이나 역 접근이 제한되는 경우
- 도로는 일부 작동하나 철도 하차 후 병목이 생기는 경우

## 7. 환승과 현장 처리 시간

복합 수송은 환승 시간이 성과를 크게 바꾼다. 현재는 기본 환승 시간이 포함되어 있지만, 실제 환승은 단순 보행 시간이 아니다.

현실적인 환승 시간에는 다음이 포함된다.

- 버스 하차
- 인원 집결
- 인원 확인
- 장비 휴대 또는 적재
- 플랫폼 이동
- 열차 대기
- 열차 탑승 통제
- 열차 하차
- 하차 후 재집결
- 마지막 버스 배정
- 목적지 하차 및 인계

특히 1,000명 규모에서는 환승 시간이 인원 수에 따라 증가할 가능성이 크다. 따라서 환승 시간을 고정값 하나로 두기보다, 기본 시간과 인원 규모에 따른 추가 시간을 함께 고려해야 한다.

현실화된 모형은 다음을 구분해야 한다.

- 소규모 인원 환승
- 1,000명 규모 환승
- 장비 휴대 인원 환승
- 플랫폼 또는 대기 공간이 부족한 환승
- 지휘통제 절차가 강화된 환승

복합 수송이 실제로 유리하려면 철도 구간의 빠른 이동이 환승 손실을 상쇄해야 한다. 따라서 환승 시간 보정은 결론의 신뢰도를 좌우한다.

## 8. 평가 지표의 현실화

현재 지표는 완료시간, 도착률, 미도착 인원, 자원 효율을 포함한다. 현실적인 판단에는 여기에 몇 가지 지표가 더 필요할 수 있다.

추가로 고려할 지표는 다음과 같다.

| 지표 | 의미 |
|---|---|
| 목표 시간 내 도착률 | 작전상 요구 시간 안에 도착한 비율 |
| 지연 인원 분포 | 전체가 아니라 몇 명이 얼마나 늦었는지 |
| 첫 도착 시간 | 선발대가 언제 도착하는지 |
| 50%, 80%, 95% 도착 시간 | 병력 축적 속도 |
| 차량 총 운행시간 | 차량과 운전 인력 부담 |
| 열차 총 운행시간 | 철도 자원 부담 |
| 환승 대기 총량 | 환승 병목 정도 |
| 취약 링크 기여도 | 어느 구간이 결과를 망치는지 |
| 실패 시나리오별 worst case | 평균이 아니라 나쁜 상황에서의 성과 |

논문이나 정책 보고서에서는 평균 완료시간만 제시하면 부족하다. 실제 운용에서는 “대부분이 언제 도착하는가”, “몇 명이 끝까지 못 오는가”, “어떤 구간이 위험한가”가 더 중요할 수 있다.

## 9. 검증과 보정

현실적인 시뮬레이션은 구현이 돌아가는 것만으로 충분하지 않다. 결과가 실제와 어느 정도 맞는지 검증해야 한다.

필요한 검증 방식은 다음과 같다.

### 내부 검증

- 같은 입력이면 같은 결과가 재현되는지 확인
- 극단 조건에서 결과가 상식적으로 움직이는지 확인
- 차량 수를 늘리면 완료시간이 줄어드는지 확인
- 도로 용량을 줄이면 지연이 늘어나는지 확인
- 환승 시간을 늘리면 복합 수송이 불리해지는지 확인
- 장애 확률이 커질수록 미도착 인원이 늘어나는지 확인

### 외부 검증

- 실제 또는 준실제 도로 통행시간과 비교
- 유사 거리 버스 이동시간과 비교
- 철도 운행시간과 대기시간의 현실성 확인
- 1,000명 규모 탑승·하차·환승 처리 시간이 가능한지 전문가 검토
- 실제 수송계획 담당자 또는 교통 전문가의 시나리오 검토

### 민감도 검증

결과가 특정 가정 하나에 너무 의존하면 논문 설득력이 약해진다. 따라서 다음 요소를 바꾸며 결론이 유지되는지 확인해야 한다.

- 차량 대수
- 배차 간격
- 회차 시간
- 철도 운행 간격
- 철도 수송 능력
- 환승 시간
- 도로 혼잡 수준
- 장애 확률
- 장애 방식
- 마지막 도로 구간 우회 가능성
- 인원 규모
- 집결 지연 분포

## 10. 논문 제출 수준을 위한 최소 기준

논문 제출을 목표로 한다면, 최소한 다음 기준은 충족해야 한다.

### 필수 기준

- 네트워크 구조가 왜 타당한지 설명되어야 한다.
- 모든 주요 모수의 근거가 표로 정리되어야 한다.
- 최소 2개 이상의 네트워크 구조를 비교해야 한다.
- 버스 단독과 복합 수송이 불공정하게 유리하거나 불리하지 않도록 경로 중복성을 점검해야 한다.
- 결과는 평균뿐 아니라 신뢰구간 또는 반복 실험 변동성을 함께 제시해야 한다.
- 미도착 인원이 있는 경우 단순 완료시간만으로 결론을 내리지 않아야 한다.
- 민감도 분석을 통해 결론이 어떤 가정에 민감한지 보여야 한다.
- 모형 한계가 명확히 서술되어야 한다.
- 실험을 재현할 수 있도록 입력값과 난수 설정이 기록되어야 한다.

### 권장 기준

- 실제 지도 기반 네트워크 사용
- 실제 통행시간 또는 공개 교통자료 기반 보정
- 철도 운행 조건의 현실성 검토
- 환승 시간에 대한 별도 근거 제시
- 전문가 검토 또는 문헌 기반 모수 보정
- 시나리오별 취약 구간 분석
- 정책적 함의와 모형 한계의 분리

## 10-A. SCI급 논문으로 발전시키기 위한 framing

현재 아이디어는 논문 소재가 될 수 있다. 그러나 현재 상태 그대로 높은 영향력의 SCI급 저널에 바로 제출할 수준은 아니다. 현재 프로젝트의 강점은 단순한 평균시간 비교를 넘어 유한 차량, 출발 정책, 철도 운행 간격, 환승 지연, 도로 혼잡, 도로 차단·용량 감소, 미도착 인원, 자원 효율을 함께 다룬다는 점이다. 반면 약점은 실세계 네트워크와 입력값 근거, 검증 구조가 아직 충분하지 않다는 점이다.

따라서 논문 framing은 다음처럼 잡는 것이 좋다.

> 공개 지리정보와 이산사건 미시 시뮬레이션을 결합하여, 네트워크 교란과 차량 제약 조건에서 대규모 인원 동원 수송의 multimodal resilience를 평가하는 재사용형 의사결정 프레임워크

피해야 할 framing은 다음이다.

> 현재 시뮬레이션이 실제 세계에서 특정 수송 방식의 절대적 우위를 증명한다.

좋은 연구 질문은 다음과 같다.

- 어떤 교란 조건과 자원 제약에서 철도-버스 복합 수송이 버스 단독 수송보다 유리해지는가?
- 복합 수송의 성과를 무너뜨리는 핵심 병목은 철도 자체인가, 철도 접근 구간인가, 마지막 도로 구간인가?
- 차량 부족, 환승 지연, 도로 장애 중 어떤 요인이 제한시간 내 도착률과 미도착 인원을 가장 크게 좌우하는가?
- 공개 지도 기반 지역 확장형 pipeline으로 여러 지역의 수송 복원력을 같은 기준으로 비교할 수 있는가?

이 방향은 군사적 세부사항을 과도하게 노출하지 않으면서도, 교통 복원력, 교란된 물류, 재난·비상 수송, multimodal network reliability 분야로 일반화할 수 있다.

### 심사자가 공격할 핵심 약점

SCI급 심사에서 가장 먼저 지적될 가능성이 높은 부분은 다음이다.

| 예상 지적 | 보강 방향 |
|---|---|
| 네트워크가 현실을 대표하는가? | OSM 기반 준현실 네트워크, 권역화된 목적지, 복수 지역 사례 적용 |
| 파라미터 근거가 무엇인가? | 입력값별 public data, literature, expert assumption, sensitivity-only 분류표 작성 |
| 검증은 어떻게 했는가? | 공개 자료 기반 통행시간 검증, 철도 headway 검토, queue/fleet sanity check |
| 기여가 시뮬레이터인가 결과인가 방법론인가? | 지역 재사용형 의사결정 프레임워크와 조건부 resilience rule로 정리 |
| 군사 사례라 일반화가 어렵지 않은가? | disrupted regional personnel movement 또는 emergency mobilization transport로 일반화 |

### 논문 claim의 적절한 형태

최종 논문 claim은 절대 우열이 아니라 조건부 규칙이어야 한다.

좋은 claim 예시는 다음과 같다.

- 버스 단독 수송은 직접 도로 접근성이 유지될 때 시간상 유리하다.
- 철도-버스 복합 수송은 차량 부족이나 장거리 도로 부담이 커질 때 자원 효율 측면에서 장점이 커질 수 있다.
- 복합 수송의 취약성은 철도 본선보다 철도 접근 도로와 마지막 도로 구간에서 더 크게 나타날 수 있다.
- 환승 지연과 라스트마일 병목이 일정 수준을 넘으면 철도의 대량 수송 이점이 사라진다.
- 평균 완료시간보다 완료확률, 미도착 인원, 95% 도착시간, 취약 링크 기여도가 더 중요한 의사결정 지표다.

## 10-B. 현실적인 3개월 보강 계획

논문화 가능성을 높이려면 다음 순서가 현실적이다.

### 1개월 차: 준현실 네트워크 prototype

- OSMnx 기반 도로망 추출 prototype을 만든다.
- 민감 목적지는 정확 좌표가 아니라 권역 중심점 또는 synthetic node로 처리한다.
- 철도역 후보와 접근도로를 네트워크에 연결한다.
- partridge 또는 gtfs_kit 기반으로 최소 철도 입력 구조를 만든다.
- 지역 입력 schema를 정의한다.

### 2개월 차: 모수 근거와 검증 구조

- 현재 시뮬레이터를 준현실 네트워크 위에서 실행한다.
- 모든 주요 입력값에 출처 등급을 붙인다.
- 도로 통행시간, 철도 운행 간격, 차량 회차, 환승 시간, queue behavior가 현실적 범위인지 검토한다.
- 민감도-only 가정과 근거 있는 값을 분리한다.
- baseline, optimistic, degraded 조건을 나눈다.

### 3개월 차: 논문용 실험 패키지

- SALib 기반 global sensitivity analysis를 추가한다.
- 정책 대안을 확장한다.
- 최소한 다음 대안을 비교한다.
  - 버스 단독
  - 철도-버스 복합
  - 라스트마일 보강 복합 수송
  - 철도 지연 조건
  - 차량 부족 조건
  - staggered dispatch 또는 adaptive routing 조건
- 결과는 평균시간보다 completion probability, censored personnel, 95th-percentile arrival time, resource efficiency, bottleneck attribution 중심으로 정리한다.
- 재현성 패키지를 만든다.

## 11. 우선순위별 작업 계획

현실화 작업은 한 번에 모두 할 필요는 없다. 다음 순서가 가장 합리적이다.

### 1단계: 현재 모형의 가정 정리

목표는 현재 결과가 어떤 가정 위에서 나온 것인지 투명하게 만드는 것이다.

- 현재 네트워크 노드와 링크의 의미를 설명한다.
- 각 통행시간과 용량이 어떤 현실 상황을 대표하는지 적는다.
- 차량 수, 배차 간격, 철도 용량, 환승 시간의 가정 근거를 정리한다.
- 현재 결과가 현실 예측이 아니라 조건부 비교임을 명확히 한다.

이 단계만 해도 보고서 신뢰도는 크게 올라간다.

### 2단계: 준현실 네트워크로 확장

목표는 실제 지도를 완전히 쓰기 전, 현실적인 구조를 더 많이 반영하는 것이다.

- 출발지, 철도 접근 거점, 철도 도착 거점, 목적지 권역을 더 현실적으로 설정한다.
- 버스 단독 경로와 복합 수송 경로의 우회 가능성을 공정하게 맞춘다.
- 마지막 도로 구간을 단일 링크가 아니라 여러 대체 경로로 표현한다.
- 철도 접근 도로의 병목을 별도 구간으로 분리한다.
- 공개 지도 또는 오픈소스 네트워크 생성 도구를 시험 적용한다.
- 지역 입력값만 바꾸면 다른 지역에서도 같은 실험을 돌릴 수 있는 구조를 만든다.

이 단계는 논문 초안 수준에서 가장 비용 대비 효과가 크다.

### 3단계: 모수 보정

목표는 주요 입력값을 임의값이 아니라 근거 있는 값으로 바꾸는 것이다.

- 도로 통행시간 보정
- 도로 용량 보정
- 차량 가용성 보정
- 환승 시간 보정
- 철도 운행 간격 보정
- 장애 확률과 용량 저하 수준 보정

모든 값을 완벽하게 실제 자료로 맞추기 어렵다면, 기준값과 범위를 함께 제시하면 된다.

### 4단계: 민감도 분석 확대

목표는 결론이 얼마나 튼튼한지 확인하는 것이다.

- 낙관, 기준, 악화 조건을 나눈다.
- 차량 부족 조건을 추가한다.
- 철도 지연 조건을 추가한다.
- 마지막 도로 구간 장애 조건을 강화한다.
- 환승 시간이 길어지는 조건을 추가한다.
- 인원 규모를 바꿔 본다.

이 단계에서 “어떤 조건이면 복합 수송이 유리해지는지” 또는 “어떤 조건이면 버스 단독이 취약해지는지”를 더 명확히 설명할 수 있다.

### 5단계: 논문용 검증 패키지 정리

목표는 결과를 독자가 신뢰하고 재현할 수 있게 만드는 것이다.

- 입력값 표
- 시나리오 표
- 반복 횟수와 난수 설정
- 평가 지표 정의
- 결과 요약표
- 주요 그림
- 민감도 분석 결과
- 한계와 위협 요인
- 재현성 설명

## 12. 최종적으로 필요한 산출물

현실에 가까운 시뮬레이션으로 인정받으려면 다음 산출물이 필요하다.

| 산출물 | 목적 |
|---|---|
| 네트워크 설명 문서 | 네트워크가 어떤 현실 구조를 대표하는지 설명 |
| 입력 모수 표 | 모든 주요 입력값과 근거 정리 |
| 시나리오 설계표 | 어떤 조건을 비교했는지 정리 |
| 검증 결과 | 모형이 상식적이고 일관되게 작동하는지 확인 |
| 민감도 분석 결과 | 결론이 어떤 가정에 민감한지 확인 |
| 결과 요약표 | 핵심 성과를 수치로 비교 |
| 해석 가이드 | 평균, 미도착, 자원 효율을 어떻게 함께 볼지 설명 |
| 한계 정리 | 현실 예측으로 과장하지 않도록 경계 |

## 13. 현재 프로젝트에 바로 적용할 수 있는 개선 방향

현재 프로젝트 기준으로 가장 먼저 하면 좋은 작업은 다음이다.

1. 현재 추상 네트워크의 각 노드와 링크가 현실에서 무엇을 대표하는지 문서화한다.
2. 버스 단독과 복합 수송의 우회 가능성이 공정한지 다시 점검한다.
3. 마지막 도로 구간을 더 현실적으로 세분화한다.
4. 철도 접근 도로와 철도 하차 후 도로를 별도 병목으로 분석한다.
5. 환승 시간을 단일값이 아니라 최소·기준·악화 조건으로 나눈다.
6. 차량 대수와 회차 시간을 여러 수준으로 바꿔 본다.
7. 철도 지연 또는 철도 운행 제한 시나리오를 추가한다.
8. 현재 결과를 평균 중심이 아니라 완료율, 미도착 인원, 자원 효율 중심으로 다시 요약한다.
9. 모든 주요 입력값에 “자료 기반”, “전문가 판단”, “민감도 가정” 중 하나의 출처 유형을 붙인다.
10. 논문용으로는 결과 표와 그림마다 “어떤 가정에서 나온 결과인지”를 명확히 표시한다.

## 14. 제약 조건 안에서 현실성을 높이는 방법

현실에 가까운 시뮬레이션은 모든 현실 정보를 그대로 넣는다는 뜻이 아니다. 실제 연구에서는 자료 접근성, 보안, 계산 비용, 검증 가능성이라는 제약이 있다. 중요한 것은 이 제약을 인정하고, 그 안에서 가장 덜 왜곡되는 방식으로 모형을 설계하는 것이다.

### 적용해야 할 제약

현실화를 추진할 때 다음 제약을 명시적으로 두는 것이 좋다.

| 제약 | 이유 | 대응 방식 |
|---|---|---|
| 민감 위치 보호 | 실제 군 목적지나 취약 시설 노출 방지 | 좌표 대신 권역, 가상 노드, 거리대 사용 |
| 공개 자료 우선 | 재현성과 검증 가능성 확보 | 공개 지도, 공개 교통자료, 문헌값 우선 사용 |
| 지역 재사용성 | 송파구 외 다른 지역에도 적용 | 지역 설정과 시뮬레이션 로직 분리 |
| 모수 불확실성 인정 | 실제 값이 불명확한 경우가 많음 | 단일값보다 범위와 민감도 분석 사용 |
| 계산 비용 제한 | 실제 대규모 교통 시뮬레이션은 비용이 큼 | 중간 해상도 네트워크와 반복 실험 균형 |
| 결과 과장 방지 | 정책 예측으로 오해될 위험 | 조건부 결과와 한계 명시 |

### 제약 안에서 현실에 가깝게 만드는 원칙

다음 원칙을 따르면 완전한 실제 자료가 없어도 연구 신뢰도를 높일 수 있다.

1. 실제 지도를 쓰되 민감한 지점은 권역화한다.
2. 도로망은 자동 추출하되, 주요 병목과 마지막 접근로는 수동 검토한다.
3. 도로 속도와 용량은 공개 자료, 도로 등급, 문헌값으로 1차 추정한다.
4. 불확실한 값은 하나로 고정하지 말고 낙관·기준·악화 범위로 둔다.
5. 수송 방식 간 비교가 불공정하지 않도록 우회 가능성을 맞춘다.
6. 평균 결과보다 미도착 인원과 취약 구간을 더 중요하게 본다.
7. 결과를 특정 지역 하나의 결론으로 닫지 않고, 지역별 반복 가능한 pipeline으로 설계한다.

### 권장 현실화 수준

현실성과 실현 가능성을 함께 고려하면 다음 세 단계가 적절하다.

| 수준 | 설명 | 사용 목적 |
|---|---|---|
| Level 1: 대표 네트워크 | 현재처럼 구조를 단순화한 네트워크 | 개념 검증, 모델 동작 확인 |
| Level 2: 준현실 네트워크 | 공개 지도 기반 도로망과 권역화된 목적지 사용 | 논문 초안, 정책 대안 비교 |
| Level 3: 보정 네트워크 | 실제 통행시간, 용량, 철도, 차량 가용성 보정 | 논문 최종본, 실무 검토 |

현재 프로젝트는 Level 1과 Level 2 사이로 발전할 준비가 된 상태다. 다음 목표는 완전한 Level 3이 아니라, 먼저 Level 2를 안정적으로 만드는 것이다.

## 15. 지역 확장형 시뮬레이션 pipeline 제안

송파구 외 다른 지역까지 확장하려면 다음 pipeline이 필요하다.

1. 지역 선택
2. 공개 지도 기반 도로망 추출
3. 철도 거점 후보 추출
4. 출발지와 목적지 권역 설정
5. 버스 단독 후보 경로 생성
6. 철도-버스 복합 후보 경로 생성
7. 도로·철도·환승 모수 보정
8. 장애 및 혼잡 시나리오 생성
9. 반복 시뮬레이션 실행
10. 완료시간, 미도착 인원, 자원 효율, 취약 구간 비교
11. 지역별 결과를 같은 형식으로 저장

이 pipeline이 있으면 특정 지역 하나에 맞춘 일회성 코드가 아니라, 여러 지역을 같은 기준으로 비교하는 연구 도구가 된다.

가장 중요한 설계 원칙은 다음이다.

- 지역 데이터는 바뀌어도 시뮬레이션 엔진은 그대로 둔다.
- 시나리오 조건은 지역별 파일로 분리한다.
- 네트워크 생성과 실험 실행을 분리한다.
- 결과 지표는 모든 지역에서 동일하게 유지한다.
- 민감한 좌표는 보고서 출력 단계에서 권역화하거나 익명화한다.

## 결론

현실에 가까운 시뮬레이션을 만들기 위해 가장 중요한 것은 더 복잡한 코드가 아니라, 네트워크와 입력값의 근거를 강화하는 것이다.

현재 모형은 수송 방식의 구조적 차이를 비교하는 데 충분한 출발점이다. 그러나 실제 정책 판단이나 논문 제출 수준으로 가려면 실제 또는 준실제 네트워크, 모수 보정, 장애 시나리오 근거, 환승 시간 검증, 민감도 분석, 재현성 문서화가 필요하다.

가장 현실적인 다음 단계는 “완전한 실제 도로망 구축”보다 먼저, 현재 추상 네트워크를 어떤 현실 구조의 대표값으로 볼 수 있는지 정리하고, 공개 지도 기반 네트워크 생성 방식을 시험 적용하는 것이다. 그 다음 주요 입력값의 근거와 민감도 범위를 명확히 하고, 지역 입력값만 바꾸면 다른 지역에서도 같은 방식으로 실험할 수 있는 pipeline으로 발전시키는 것이 가장 안정적이다.
